Git Control and Github and Branching

This web site is build keeping in mind the best practices of git version control. First a repository at Github created including a readme.md file giving a well defined name to highlight the purpose of the project. Once the repository is created, a new project folder is created on the IDE and the remote repository is cloned using the following command git clone https://github.com/NeilMoras/Http5201_Assignment_4.git

After the project is cloned, it is automatically linked to the repositories main branch by default. Since I am the only person pulling and pushing code from the remote repo, no branch is created and the project is pushed directly to the main branch after update. If the same project is contributed by one or more developer, it is a good practice to create a separate branch and then pushing your code to your new branch. once you push your code to the new branch, there is an option to create a pull request which will pull all the code from the main branch to your branch.You can tag your fellow developer working on the same repo to review your code. If there are any conflicts, it will be reflected and github will display the conflict and allow you to fix the conflict and once fixed,its best for your fellow developer to review it before merging it to the main branch. This method will help your code on your computer to be unaffected by any conflicts as your branch will handle the conflicts remotely and merge it.

Even if I am the sole contributor working on the project, it is a good habit to always pull after making a commit as that will create a muscle memory

Here are the following steps from cloning the repo and pushing your code to the repository

  1. git clone https://github.com/NeilMoras/Http5201_Assignment_4.git
  2. git status (provides information on the current branch and also any uncommited changes)
  3. git Add . (add all files) => git commit -m "first commit" OR git commit -am "first commit"
  4. git pull(fix and conflict after pull)
  5. git push
